KEY POINTS:
-----------
1. db_cmd.py does NOT import qunix_cpu.py
2. Communication is ONLY via database tables
3. qunix_cpu.py runs as SEPARATE process/daemon
4. Both poll their respective tables
5. No Python object sharing = no conflicts!

TABLES USED:
-----------
Web Terminal (Flask/db_cmd):
- terminal_sessions
- terminal_input  
- terminal_output

CPU Command Queue (CPU integration):
- cpu_command_queue (db_cmd writes, qunix_cpu reads)
- cpu_command_results (qunix_cpu writes, db_cmd reads)
- sys_commands (command registry)

HOW db_cmd v17.0 WORKS:
-----------------------
1. Poll terminal_input for unprocessed commands
2. Check if command is in sys_commands (CPU command)
3. If CPU command:
   - Write to cpu_command_queue table
   - Poll cpu_command_results table for response
   - Write response to terminal_output
4. If basic command:
   - Execute directly
   - Write response to terminal_output

HOW qunix_cpu WORKS:
-------------------
1. Run as: python3 qunix_cpu.py /path/to/qunix_leech.db
2. Polls cpu_command_queue for new commands
3. Executes commands using CPU execution units
4. Writes results to cpu_command_results
5. Runs continuously as daemon

WELCOME MESSAGE LOOP FIX:
------------------------
- Welcome message ONLY written to terminal_output (by Flask)
- NEVER written to terminal_input
- If detected in input, immediately blocked
- No auto-execution possible

NAMESPACE FIX:
-------------
- All terminal tables use session_id as namespace
- No "conceptual.terminal" vs "sim" vs "chat" collision
- Each web session gets unique session_id
- Clean separation
